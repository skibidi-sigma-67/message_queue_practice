# Стратегия поведения при переполнении — Блокирующая очередь с ограниченной ёмкостью

## Контекст

В проекте нужно реализовать 5 разных стратегий очереди. Одной из них является блокирующая очередь с ограниченной ёмкостью. Такое поведение необходимо для сценариев, где принципиально важно обеспечить обработку каждого элемента, а потеря данных недопустима. При этом необходимо жестко контролировать потребление памяти, чтобы система не исчерпала ресурсы в случае, если производитель генерирует данные быстрее, чем их успевает обрабатывать потребитель.

## Решение

Мы реализуем блокирующую очередь со следующим поведением:

* Для хранения сообщений используется контейнер `std::deque` , так как он удобен для FIFO-очереди.


* Очередь имеет жестко заданную максимальную ёмкость.
* Блокирующая очередь блокирует producer, если очередь заполнена.


* Для предотвращения активного ожидания применяются `std::condition_variable`.


* В рамках этой стратегии гарантируется доставка всех добавленных сообщений.


* Для защиты общих данных и того, чтобы избежать race condition, все операции с очередью защищаются с помощью `std::mutex`.



## Последствия

**Плюсы:**

* Гарантируется доставка сообщений без риска их сброса или потери.
* Расход оперативной памяти строго ограничен заданным размером буфера.
* Естественное балансирование нагрузки: если consumer не справляется, producer автоматически приостанавливает свою работу, не перегружая систему.

**Минусы:**

* Поток-производитель неизбежно блокируется в ожидании свободного места, что может привести к простою, если принимающая сторона работает слишком медленно или зависла.


* Возможны дополнительные накладные расходы на контекстные переключения и синхронизацию потоков через блокировки, что может снижать пиковую пропускную способность по сравнению с неблокирующими реализациями.
