### Стратегия поведения при переполнении — Кольцевая очередь со сбросом самого старого сообщения

**Контекст**
В рамках проекта необходимо реализовать пять различных вариантов поведения потокобезопасной очереди сообщений. Одной из таких стратегий является кольцевая очередь со сбросом самого старого сообщения. Такое поведение полезно там, где важны свежие данные, а старые уже не имеют большой ценности. К таким сценариям относятся работа с телеметрией, метриками, данными с датчиков и потоковыми обновлениями состояния.

**Решение**
Мы реализуем кольцевую очередь со следующим поведением:

* Элементы хранятся в буфере фиксированного размера.

* Когда буфер заполняется, очередь не блокирует поток-производитель (producer), а жертвует старыми данными.

* Новые сообщения все равно принимаются, но самое старое (еще не прочитанное) сообщение удаляется.

* Для сбора метрик внутри очереди необходимо считать количество сброшенных (dropped) сообщений.

* Чтобы внешний код понимал, что произошло при переполнении, операция `Push` будет возвращать не просто `void`, а специальный статус `PushStatus`

* Во избежание состояний гонки (race condition) при конкурентном доступе, все общие данные, такие как контейнер, счетчики и индексы кольцевого буфера, будут защищаться с помощью `std::mutex`.

**Последствия**

**Плюсы:**

* Поток-производитель почти не блокируется.

* Используемая память жестко ограничена заранее.

* Система не зависает при всплесках нагрузки.

**Минусы:**

* Часть сообщений неизбежно теряется при переполнении буфера.

* Требуется дополнительная логика — нужно считать количество сброшенных сообщений.

* Данный подход категорически не подходит для систем, где каждое сообщение обязательно должно быть обработано.
