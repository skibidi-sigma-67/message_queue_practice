# Стратегия поведения при переполнении — Кольцевая очередь с блокировкой писателя

## Контекст

В проекте нужно реализовать 5 разных стратегий очереди. Одной из них является кольцевая очередь с блокировкой писателя (writer-blocking). Такое поведение необходимо там, где потеря сообщений недопустима, а допустимо временное замедление потока-производителя. К таким сценариям относятся пулы потоков, конвейеры обработки данных, системы логирования с гарантированной записью и producer-consumer архитектуры, где важна целостность потока данных.

## Решение

Мы реализуем кольцевую очередь со следующим поведением:

* Для хранения сообщений используется кольцевой буфер на основе `std::vector` фиксированного размера, выделяемого один раз при создании очереди. Позиции головы и хвоста и текущий размер отслеживаются индексами.


* Очередь имеет жестко заданную максимальную ёмкость.


* Когда буфер заполнен, очередь блокирует producer до тех пор, пока consumer не заберёт хотя бы одно сообщение и не освободит место.


* Для предотвращения активного ожидания применяются два `std::condition_variable` — отдельно для ожидания свободного места и для ожидания появления сообщений.


* В рамках этой стратегии гарантируется доставка всех добавленных сообщений, порядок сообщений строго сохраняется (FIFO).


* Для защиты общих данных и того, чтобы избежать race condition, все операции с очередью защищаются с помощью `std::mutex`.


* Для корректного завершения работы предусмотрен метод `Close`, который помечает очередь закрытой и будит все заблокированные потоки.



## Последствия

**Плюсы:**

* Гарантируется доставка сообщений без риска их сброса или потери.
* Расход оперативной памяти строго ограничен заданным размером буфера.
* Естественное балансирование нагрузки: если consumer не справляется, producer автоматически приостанавливает свою работу, не перегружая систему.

**Минусы:**

* Поток-производитель неизбежно блокируется в ожидании свободного места, что может привести к простою, если принимающая сторона работает слишком медленно или зависла.


* Если единственный consumer завершился, а очередь заполнена, producer будет ждать бесконечно — для аварийного завершения необходимо использовать `Close`.


* Подход не подходит для real-time систем, где producer должен работать с гарантированной скоростью, а также для сценариев, где допустима потеря устаревших данных (потоковое видео, телеметрия).
