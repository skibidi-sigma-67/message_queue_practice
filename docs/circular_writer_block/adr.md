### Стратегия поведения при переполнении — Кольцевая очередь с блокировкой писателя

**Контекст**
В рамках проекта необходимо реализовать пять различных вариантов поведения потокобезопасной очереди сообщений. Одной из таких стратегий является кольцевая очередь с блокировкой писателя (writer-blocking). Такое поведение полезно там, где потеря сообщений недопустима, а допустимо временное замедление потока-производителя. К таким сценариям относятся пулы потоков, конвейеры обработки данных, системы логирования с гарантированной записью и producer-consumer архитектуры, где важна целостность потока данных.

**Решение**
Мы реализуем кольцевую очередь со следующим поведением:

* Элементы хранятся в буфере фиксированного размера (`std::vector<Message>`), выделяемом один раз при создании очереди.

* Когда буфер заполняется, очередь блокирует поток-производитель (producer) до тех пор, пока поток-потребитель (consumer) не заберёт хотя бы одно сообщение и не освободит место.

* Ни одно сообщение не теряется — каждое принятое сообщение гарантированно будет доставлено потребителю.

* Для синхронизации используются два `std::condition_variable`: `not_full_` (на которой ожидает producer при заполненной очереди) и `not_empty_` (на которой ожидает consumer при пустой очереди).

* Операция `Push` будет возвращать специальный статус `PushStatus`, позволяющий внешнему коду различать успешную запись (`Success`) и попытку записи в закрытую очередь (`Closed`).

* Во избежание состояний гонки (race condition) при конкурентном доступе, все общие данные, такие как контейнер, счётчики и индексы кольцевого буфера, будут защищаться с помощью `std::mutex`.

* Для корректного завершения работы предусмотрен метод `Close`, который устанавливает флаг закрытия и будит все заблокированные потоки через `notify_all`.

* Для сбора метрик внутри очереди необходимо считать количество случаев блокировки producer (`push_blocked_count`), что позволяет оценить, насколько часто очередь переполняется.

**Последствия**

**Плюсы:**

* Гарантируется доставка каждого сообщения — ни одно сообщение не теряется.

* Используемая память жёстко ограничена заранее заданной ёмкостью буфера.

* Обеспечивается естественный back-pressure: если consumer не успевает обрабатывать сообщения, producer автоматически замедляется.

* Порядок сообщений строго сохраняется (FIFO).

**Минусы:**

* Поток-производитель может блокироваться на неопределённое время, если consumer работает медленно.

* Если единственный consumer завершился или заблокирован, а очередь заполнена, все producer будут ждать бесконечно (необходимо использовать `Close` для аварийного завершения).

* Данный подход не подходит для real-time систем, где producer должен работать с гарантированной скоростью, а также для сценариев, где допустима потеря устаревших данных (потоковое видео, телеметрия).
