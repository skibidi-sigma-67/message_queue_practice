# Стратегия поведения очереди — Lock-free очередь Michael-Scott

## Контекст

В проекте нужно реализовать 5 разных стратегий очереди. Одной из них является lock-free очередь на основе алгоритма Michael-Scott. Такое поведение необходимо для сценариев, где с очередью одновременно работают несколько producer и consumer потоков, а использование глобальной блокировки может стать узким местом и снизить пропускную способность.

Lock-free очередь должна обеспечивать потокобезопасную запись и чтение сообщений без явного использования `std::mutex` для защиты основной структуры данных. При этом очередь должна реализовывать общий интерфейс `BaseQueue`, поддерживать добавление сообщений, неблокирующее чтение, чтение с ожиданием, закрытие очереди, получение текущего размера и статистики.

## Решение

Мы реализуем lock-free очередь со следующим поведением:

* Для хранения сообщений используется односвязный список узлов. В начале списка находится dummy-узел, который упрощает операции добавления и удаления.

* Очередь является неограниченной по емкости: при переполнении сообщения не сбрасываются, а новые узлы выделяются динамически.

* Для синхронизации producer и consumer потоков используются атомарные указатели `head` и `tail`, а также CAS-операции через `compare_exchange_weak` или `compare_exchange_strong`.

* Операция `Push` добавляет новый узел в конец очереди с помощью CAS на поле `next` текущего хвоста. После успешного добавления поток пытается продвинуть `tail` на новый узел.

* Операция `TryPop` читает следующий узел после `head`. Если очередь не пуста, поток пытается CAS-операцией продвинуть `head` вперед и вернуть сообщение из удаляемого логического узла.

* Если один поток видит, что `tail` отстал от фактического конца списка, он помогает продвинуть `tail`. Это является частью алгоритма Michael-Scott и позволяет системе продолжать работу без ожидания конкретного потока.

* Метод `WaitPop` реализуется поверх `TryPop`: сначала поток пытается извлечь сообщение без блокировки, а если очередь пуста, ожидает уведомления через атомарный счетчик и механизм `std::atomic::wait` / `std::atomic::notify_one`.

* Метод `Close` переводит очередь в закрытое состояние. После закрытия новые сообщения не принимаются, `Push` возвращает `PushStatus::kError`, а ожидающие consumer потоки пробуждаются.

* Для статистики используются атомарные счетчики: количество успешных добавлений, успешных чтений, сброшенных сообщений, ошибочных операций и текущий размер. Метод `GetStats` возвращает обычный `QueueStats` как снимок этих атомарных счетчиков на момент вызова.

* Структура `QueueStats` остается неатомарной, потому что она является интерфейсным типом результата. Атомарность нужна только внутри lock-free реализации, где счетчики обновляются конкурентно.

* Для безопасного управления памятью необходимо учитывать, что узел нельзя удалять сразу после успешного `pop`: другой поток мог ранее прочитать указатель на этот узел. Решение: использование hazard указателей.

## Последствия

**Плюсы:**

* Основные операции записи и чтения выполняются без глобального `std::mutex`, поэтому при высокой конкуренции очередь может показывать лучшую пропускную способность по сравнению с блокирующими реализациями.

* Очередь подходит для MPMC-сценариев, где несколько producer и consumer потоков одновременно добавляют и читают сообщения.

* Отсутствие блокировки основной структуры данных снижает риск простоя всех потоков из-за одного медленного или остановленного потока.

* Алгоритм Michael-Scott сохраняет FIFO-порядок сообщений.

* Для добавленных сообщений не используется политика сброса, поэтому `dropout_count` для этой стратегии остается равным нулю.

**Минусы:**

* Реализация значительно сложнее блокирующих очередей на `std::mutex` и `std::condition_variable`.

* Корректное освобождение памяти в lock-free структуре требует отдельной схемы memory reclamation. Простое удаление узла сразу после `pop` может привести к use-after-free.

* Очередь не имеет жесткого ограничения по памяти, поэтому при producer, который работает намного быстрее consumer, потребление памяти может расти.

* Метод `Size` в конкурентной среде возвращает только приближенно актуальный снимок состояния, так как другие потоки могут изменить очередь сразу после чтения счетчика.

* `WaitPop` все равно требует механизма ожидания и пробуждения, потому что интерфейс `BaseQueue` поддерживает блокирующее чтение при пустой очереди.
