# ADR-006: Стратегия поведения — Приоритетная очередь на основе min-heap с сохранением FIFO-порядка при равных приоритетах

**Статус**
Предложено.

**Контекст**
В рамках проекта необходимо реализовать пять различных вариантов поведения потокобезопасной очереди сообщений. Одной из ключевых стратегий является приоритетная очередь, извлекающая элементы на основе их важности (минимального значения поля `priority`). 

Такое поведение критически важно для систем, где обработка определенных задач имеет наивысший жизненный приоритет (например, обработка аварийных сигналов в телеметрии, VIP-транзакции в банковских системах или системные прерывания). 


**Решение**
Мы реализуем приоритетную очередь со следующим поведением:
- Элементы хранятся в динамическом массиве (`std::vector`), поверх которого с помощью стандартных алгоритмов или контейнера выстраивается структура двоичной кучи (min-heap). Это обеспечивает логарифмическое время выполнения операции вставки и удаления $O(\log N)$. 

- Для обеспечения FIFO-порядка при равных приоритетах вводится внутренний монотонно возрастающий счетчик уникальных номеров (`sequence_number`). Каждое входящее сообщение `Message`  оборачивается во внутреннюю структуру узла, содержащую поле `seq_id`.  

- Компаратор кучи сначала сравнивает поле `priority`, отдавая предпочтение минимальному значению (min-heap), а при их равенстве — поле `seq_id` (первым извлекается сообщение с меньшим ID, то есть пришедшее раньше). 

- Очередь проектируется как неограниченная по емкости, поэтому потоки-производители (producers) никогда не блокируются из-за переполнения буфера.  

- Во избежание состояний гонки при конкурентном доступе, все общие данные, такие как контейнер кучи, счетчики и индексы, будут защищаться с помощью `std::mutex`.   

- Потоки-потребители (consumers) блокируются на `std::condition_variable` (`not_empty_`), если очередь пуста, что предотвращает активное ожидание (busy waiting).  

- Операция `Push` принимает объект `Message` и возвращает специальный статус `PushStatus`, позволяющий внешнему коду различать успешную запись (`Success`) и попытку записи в закрытую очередь (`Closed`).  

- Для корректного завершения работы предусмотрен метод `Close`, который устанавливает флаг закрытия и будит все заблокированные потоки через `notify_all`. После закрытия новые сообщения через `Push` не принимаются.  

- Для сбора базовой статистики внутри очереди используется структура `QueueStats`, которая хранит количество успешных `Push`, успешных `Pop` и текущее значение размера `Size` .  

**Последствия**

**Плюсы:**

- Абсолютная точность соблюдения приоритетов задач с гарантированным сохранением порядка FIFO для равных элементов.

- Эффективность по памяти и отличная локальность данных в кэше процессора благодаря использованию векторной кучи вместо сложных узловых структур (деревьев или списков).  

- Предсказуемая и стабильная асимптотическая сложность операций записи и чтения ($O(\log N)$).  

- Благодаря общему абстрактному интерфейсу `BaseQueue` стратегию легко тестировать и сравнивать в рамках разработанного параметризуемого бенчмарка.  

**Минусы:**

- Появляется небольшая вычислительная стоимость виртуальных вызовов из-за наследования от общего интерфейса `BaseQueue`.  

- Риск голодания низкоприоритетных сообщений: если поток высокоприоритетных задач непрерывен, сообщения с низким приоритетом могут остаться в очереди на неопределенный срок.  

- Использование одного `std::mutex` на всю структуру создает бутылочное горлышко при огромном количестве конкурирующих потоков производителей и потребителей.
