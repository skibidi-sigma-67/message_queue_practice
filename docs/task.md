# Сравнение стратегий поведения очереди сообщений в многопоточном in-memory брокере на различных паттернах нагрузки

### Аннотация
* Цель определить оптимальную стратегию поведения очереди сообщений для различных сценариев работы на основе объективных метрик, включая пропускную способность, латентность операций, использование памяти и fairness между писателями и читателями.
* Вам предстоит реализовать потокобезопасную очередь сообщений с поддержкой пяти стратегий поведения, реализовать in-memory хранение без использования файлов и баз данных, организовать сбор статистики работы очереди в конкурентной среде, разработать параметризуемый бенчмарк с различными профилями нагрузки, а также провести экспериментальные замеры и проанализировать полученные результаты.

---

## 1. Теоретическая часть
В рамках теоретической подготовки вам необходимо провести обзор предметной области по следующим направлениям.

### 1.1. Стратегии поведения очереди сообщений
* Вам предстоит самостоятельно изучить назначение, решаемые проблемы, алгоритмические особенности, а также асимптотическую сложность по памяти и времени на операции записи и чтения для следующих стратегий поведения очереди сообщений:
    * Блокирующая очередь с ограниченной ёмкостью на основе списка или дека.
    * Кольцевая очередь с политикой переполнения «сброс самого старого».
    * Кольцевая очередь с политикой переполнения «блокировка писателя».
    * Приоритетная очередь на основе min-heap с сохранением FIFO-порядка при равных приоритетах.
    * Очередь с использованием lock-free структуры Michael-Scott.
* Для поиска релевантных источников используйте Google Scholar.
* При возникновении языкового барьера допускается загрузка статьи в диалоговое окно LLM с последующим уточнением непонятных фрагментов.
* Аналогичный метод поиска и анализа информации следует применять на всех последующих этапах.

### 1.2. Профили нагрузки
* Вам необходимо изучить следующие профили нагрузки применительно к очереди сообщений:
    * Производитель-потребитель с равными темпами.
    * Продюсер намного быстрее консюмера.
    * Консюмер намного быстрее продюсера.
    * Пачечная запись с последующим чтением.
    * Случайные всплески с обеих сторон.
    * Смесь сообщений с разными приоритетами.
* Для каждого профиля определите, какие реальные сценарии использования систем очередей сообщений он моделирует.

### 1.3. Параметры системы, задаваемые пользователем
* Определите, в каких компонентах системы задаются следующие параметры: выбранная стратегия поведения очереди, максимальная ёмкость очереди для стратегий с ограничением, количество потоков-продюсеров, количество потоков-консюмеров, длительность тестирования для каждого профиля, интенсивность генерации сообщений для каждой стороны, распределение приоритетов сообщений для приоритетной очереди, а также количество повторных запусков на одну комбинацию параметров.
* Обоснуйте необходимость проведения серии экспериментов каждой комбинации и проанализируйте источники неопределенности.

### 1.4. Метрики оценки экспериментов
Вам предстоит изучить и применить следующие метрики:
* Пропускная способность в сообщениях в секунду.
* Медианная латентность и латентность девяносто девятого перцентиля для операций записи.
* Аналогичные метрики для операций чтения.
* Доля времени блокировки продюсеров и консюмеров.
* Количество сброшенных или затертых сообщений в стратегиях с ограниченной ёмкостью.
* Использование памяти.
* Fairness — соотношение обработанных сообщений между разными потоками.

### 1.5. Потокобезопасность
* Поскольку ранее на курсе вы не затрагивали многопоточность или затрагивали её в ограниченном объёме, необходимо проанализировать новые требования и риски.
* Ответьте на следующие вопросы: какие проблемы возникают при параллельном доступе к очереди сообщений, какие преимущества дают многопоточные приложения, с какими рисками, включая состояние гонки, вам предстоит столкнуться, а также какие стратегии синхронизации вы планируете применить для устранения race condition.
* Отдельно для варианта с lock-free очередью объясните, как гарантируется потокобезопасность без использования явных блокировок, например с помощью CAS-операций.

## 2. Реализация
* **Стек технологий:** С++23, стандартная библиотека, Google Test для модульного тестирования, Google Benchmark для бенчмаркинга.
* Порядок реализации следующий. Вам необходимо разработать библиотеку, реализующую потокобезопасную очередь сообщений.
* Рекомендуется начать с проектирования интерфейса, который должен поддерживать запись сообщения в queue, чтение сообщения из очереди с возможностью блокировки при отсутствии сообщений, а также получение текущего размера и статистики.
* После утверждения интерфейса напишите юнит-тесты, в том числе тесты, выявляющие состояния гонки при одновременной записи и чтении из множества потоков.
* Затем распределите задачи между участниками: написание юнит-тестов, разработка бенчмарка, реализация библиотеки.
* При реализации библиотеки предусмотрите поддержку всех пяти стратегий поведения очереди, а именно блокирующей очереди с ограниченной ёмкостью, кольцевой очереди со сбросом самого старого, кольцевой очереди с блокировкой писателя, приоритетной очереди и lock-free очереди на основе Michael-Scott.
* Также предусмотрите интерфейс для выбора стратегии пользователем.
* Наконец, разработайте бенчмарк, поддерживающий выполнение всех перечисленных профилей нагрузки.

## 3. Экспериментальная часть

### 3.1. Формулирование гипотез
* На основании теоретического обзора вы должны самостоятельно сформулировать гипотезы о том, какая стратегия поведения очереди сообщений окажется наиболее эффективной для каждого из следующих профилей нагрузки: производитель-потребитель с равными темпами, продюсер намного быстрее консюмера, консюмер намного быстрее продюсера, пачечная запись с последующим чтением, случайные всплески с обеих сторон, смесь сообщений с разными приоритетами.
* Для каждого профиля перечислите предполагаемого лидера среди стратегий и приведите теоретическое обоснование.
* Также сформулируйте гипотезу о том, в каких сценариях lock-free очередь даст выигрыш по сравнению с блокирующими реализациями, а в каких ее преимущества нивелируются.

### 3.2. Параметры экспериментов
Вам предстоит самостоятельно определить числовые параметры, с которыми будут проводиться экспериментальные замеры, включая:
* Максимальную ёмкость очереди.
* Количество потоков-продюсеров и потоков-консюмеров.
* Соотношение скоростей генерации и обработки сообщений для разных профилей.
* Размер сообщения.
* Распределение приоритетов для тестирования приоритетной очереди.
* Длительность каждого профиля.
* Количество повторных запусков на одну комбинацию параметров.

### 3.3. Проведение экспериментов и анализ данных
* Выполните эксперименты на разработанном бенчмарке.
* Для каждого профиля нагрузки составьте сводные таблицы метрик в разрезе применяемых стратегий поведения очереди.
* Проведите анализ полученных данных и проверьте, подтверждаются ли выдвинутые ранее гипотезы.
* В случае отклонения гипотез сформулируйте возможные причины.
* Обратите особое внимание на сравнение кольцевых очередей с разными политиками переполнения: при каких условиях предпочтительнее сброс самого старого, а при каких блокировка писателя.
* Также сравните блокирующую очередь с lock-free реализацией: дайте ответ на вопрос, оправдывает ли lock-free сложность реализации прирост производительности в ваших экспериментах.
* Сопоставьте результаты с гипотезами и сформулируйте итоговые выводы.

## 4. Результаты работы
* Итоговый результат работы включает репозиторий с кодом, содержащий реализацию потокобезопасной очереди сообщений со всеми пятью стратегиями, бенчмарк и набор юнит-тестов.
* Отчёт должен содержать результаты теоретического исследования, описание экспериментальной методологии, полученные экспериментальные данные и их интерпретацию.
* В отчёте также ответьте на контрольные вопросы: какова асимптотическая сложность по памяти и времени на операции записи и чтения для каждой реализованной вами структуры данных, и как в вашей реализации гарантируется потокобезопасность для каждой из пяти стратегий.
